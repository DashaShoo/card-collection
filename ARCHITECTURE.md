# Архитектура системы CardCollect

## 2.1 Структура и концепция системы

CardCollect — это веб-приложение для коллекционирования цифровых карточек. Система разработана с учётом нескольких важных принципов. Во-первых, приложение должно масштабироваться: backend спроектирован так, что можно добавлять новые серверы без потери данных пользователя. Во-вторых, безопасность — все данные передаются зашифрованными через HTTPS, пароли хранятся в защифрованном виде, а сессии пользователя хранятся на отдельном сервере (Redis). В-третьих, приложение работает одинаково на любых устройствах и браузерах. Кроме того, система настроена так, что один и тот же набор файлов может работать как на компьютере разработчика, так и на боевом сервере — нужно только изменить несколько параметров конфигурации. Наконец, весь процесс разработки и развёртывания автоматизирован: код хранится в Git, а при каждом обновлении автоматически собирается и развёртывается.

## 2.2 Технологический стек

Серверная часть приложения написана на **Node.js 18** с использованием фреймворка **Express 5**. Это популярная платформа, на которой легко писать быстрые веб-приложения.

На стороне пользователя приложение написано на **React 19** с **TypeScript**. React позволяет строить интерфейс из переиспользуемых компонентов, а TypeScript помогает избежать ошибок. Для сборки используется **Vite** (быстрый инструмент сборки), а для дизайна — **Tailwind CSS**.

Данные хранятся в **PostgreSQL 15** — надёжной базе данных, которая гарантирует сохранность информации. Для хранения сессий пользователей используется **Redis 7** — быстрое хранилище в памяти.

Приложение работает в **Docker-контейнерах**. Это означает, что приложение работает в изолированной среде, и один и тот же набор контейнеров можно запустить где угодно.

## 3. Развёртывание

### 3.1 Локальное развёртывание

Для удобства разработки все компоненты приложения упакованы в Docker-контейнеры. Docker — это технология, которая позволяет приложению работать в изолированной среде. Это означает, что приложение будет работать одинаково на компьютере разработчика, на тестовом сервере и на боевом сервере.

**Как работает Docker** — для каждого компонента (frontend, backend, база данных, Redis) создаётся свой файл Dockerfile, который описывает, как собрать приложение. Бэкенд собирается в два этапа: сначала устанавливаются все зависимости из файла package.json, потом копируются только нужные файлы в финальный контейнер. Это сделано для экономии места. Фронтенд собирается похожим образом: сначала React приложение собирается через Vite, потом готовые файлы копируются в Nginx (веб-сервер), который их обслуживает.

**Docker Compose** — это инструмент, который управляет всеми контейнерами вместе. Когда запускаешь `docker-compose up`, он:
- Запускает все 4 контейнера (frontend, backend, PostgreSQL, Redis)
- Настраивает сеть так, чтобы контейнеры могли общаться между собой
- Сохраняет данные в специальные папки на диске, чтобы при перезапуске данные не потерялись
- Проверяет здоровье сервисов (работают ли они) и автоматически перезапускает их при проблемах
- Конфигурация хранится в .env файле, поэтому можно легко менять параметры без изменения кода

**Как запустить приложение локально:**

```bash
# Клонируем проект
git clone https://github.com/YOUR_USERNAME/card-collection.git
cd card-collection

# Создаём файл с настройками
cat > .env << EOF
NODE_ENV=development
LOG_LEVEL=info
PORT=3000
DB_HOST=db
DB_PORT=5432
DB_NAME=cards
DB_USER=postgres
DB_PASSWORD=postgres
REDIS_HOST=redis
REDIS_PORT=6379
CORS_ORIGIN=http://localhost:3000
EOF

# Собираем и запускаем всё
docker-compose up --build

# Другие полезные команды:
docker-compose ps                 # Показать статус контейнеров
docker-compose logs -f backend    # Смотреть логи backend
docker-compose down               # Остановить всё
docker-compose down -v            # Остановить и удалить данные
```

После запуска приложение доступно по адресам:
- Frontend (интерфейс): `http://localhost:3000`
- Backend API (для тестирования): `http://localhost:8080`
- База данных PostgreSQL: `localhost:5432`
- Redis (кэш): `localhost:6379`

Nginx автоматически перенаправляет запросы на фронтенде к backend-у, поэтому браузер видит всё как один сервер.

**Преимущества такого подхода** — каждый разработчик работает в одинаковом окружении, нет проблем типа "у меня работает, а у тебя нет". Если нужно поменять версию базы данных или добавить новый сервис, достаточно отредактировать docker-compose.yml и перезапустить контейнеры.

### 3.2 Развёртывание на боевом сервере

Когда приложение готово к выпуску, его нужно выложить в интернет. Для этого есть несколько вариантов.

**Railway** — самый удобный способ. Эта платформа сама разберётся с docker-compose.yml и развернёт всё приложение. Она предоставляет $5 кредита в месяц, чего хватает для приложения среднего размера. Команда всего одна: `railway up`.

**Render** — полностью бесплатный вариант. Он разворачивает приложение без платежей, но контейнеры "спят" после 15 минут неактивности, так что приложение немного медленнее запускается. Подходит для демонстрации проекта.

**Собственный сервер** (DigitalOcean, Linode, AWS) — если нужна полная свобода. Ты просто покупаешь виртуальный сервер, устанавливаешь туда Docker и запускаешь свой docker-compose.yml. Это стоит от $5 в месяц и даёт полный контроль.

Подробные пошаговые инструкции для каждого варианта есть в файле `DEPLOYMENT_DOCKER_COMPOSE.md`.

## Итоги

CardCollect построена с использованием современных технологий и надёжных решений. Система использует Node.js и Express для backend, React для frontend, PostgreSQL для хранения данных и Redis для сессий. Приложение полностью контейнеризовано с помощью Docker, что гарантирует, что оно будет работать одинаково везде. Backend не привязан к конкретному серверу, поэтому его можно масштабировать, добавляя новые инстансы. Всё это позволяет разработчикам работать эффективно и быстро развивать приложение.
